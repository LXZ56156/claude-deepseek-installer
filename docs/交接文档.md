## 一、项目背景

### 1.1 产品定位

**Claude Code + DeepSeek API 本地配置服务**，面向闲鱼等技术服务场景。

买家购买的是**安装配置人工服务**，脚本作为交付品辅助自动化。

核心价值：把「需要用户运行 PowerShell 命令」降级为「解压后双击开始安装」，降低小白用户操作难度和售后沟通成本。

### 1.2 合规边界（红线）

| 我们做什么 | 我们不做什么 |
|-----------|-------------|
| Claude Code 本地安装 | 出售 Claude 账号 |
| DeepSeek API 本地配置 | 出售 API Key |
| 环境诊断和修复 | API 中转/代理服务 |
| 配置文件备份恢复 | 账号共享 |
| 纯脚本，可审计 | 破解/绕过限制 |

### 1.3 目标用户

- 想用 Claude Code 但不想折腾环境的小白
- 已有 DeepSeek API Key，想接入 Claude Code
- 在 Windows/WSL 环境下使用 VS Code（可选）
- 遇到安装配置问题需要诊断

## 三、架构概览

### 3.1 文件依赖链

```
开始安装.cmd / Start-Install.cmd
  └─ Start-Here.ps1（v1.3 主入口）
       └─ lib/bootstrap.ps1（唯一允许 dot-source 其他 lib 的文件）
            ├─ lib/logger.ps1（无依赖）
            ├─ lib/common.ps1（依赖 logger）
            ├─ lib/env-check.ps1（依赖 common + logger）
            └─ lib/config-writer.ps1（依赖 common + logger）
```

### 3.2 入口文件矩阵

| 文件 | 用途 | 用户类型 |
|------|------|---------|
| 开始安装.cmd | 懒人一键安装 | 小白（双击） |
| Start-Install.cmd | 英文备用安装 | 中文文件名异常的 Windows |
| 一键诊断.cmd | 环境诊断 | 有问题时双击 |
| Run-Diagnostics.cmd | 英文备用诊断 | 同上 |
| 恢复或卸载配置.cmd | 配置管理 | 换 Key / 恢复备份 |
| Restore-Config.cmd | 英文备用配置 | 同上 |
| Start-Here.ps1 | 总控入口 | 高级用户（命令行） |
| install.ps1 | 旧版安装脚本 | 向后兼容 |

### 3.3 库文件职责

| 文件 | 职责 |
|------|------|
| `lib/logger.ps1` | 日志初始化、彩色控制台输出、日志文件管理 |
| `lib/common.ps1` | 路径/备份/JSON/Key 脱敏/命令执行/用户交互/PATH 刷新 |
| `lib/env-check.ps1` | 系统/命令/文件/网络/DeepSeek API 检测 |
| `lib/config-writer.ps1` | DeepSeek 配置读写/合并/备份/恢复 |
| `lib/bootstrap.ps1` | 入口初始化、统一库加载 |
| `lib/deepseek-env.defaults.json` | PowerShell/WSL 共用默认 env 模板 |

### 3.4 DeepSeek 配置格式

写入 `%USERPROFILE%\.claude\settings.json` 的 `env` 字段：

```json
{
  "env": {
    "ANTHROPIC_BASE_URL": "https://api.deepseek.com/anthropic",
    "ANTHROPIC_AUTH_TOKEN": "<用户 Key>",
    "ANTHROPIC_MODEL": "deepseek-v4-pro[1m]",
    "ANTHROPIC_SMALL_FAST_MODEL": "deepseek-v4-flash",
    "ANTHROPIC_DEFAULT_OPUS_MODEL": "deepseek-v4-pro[1m]",
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "deepseek-v4-pro[1m]",
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "deepseek-v4-flash",
    "CLAUDE_CODE_SUBAGENT_MODEL": "deepseek-v4-flash",
    "CLAUDE_CODE_EFFORT_LEVEL": "max"
  }
}
```

## 四、遇到的问题和修复记录

### 4.1 PowerShell 5.1 中文乱码（P0）

**问题**: .ps1 文件使用 UTF-8 without BOM。Windows PowerShell 5.1 在中文 Windows 上按 GBK 读取，中文字符变成乱码，触发 ParserError（缺少右括号、意外的 }、Try 缺少 Catch/Finally）。

**修复** (`3fb9768`):
- 全部 11 个 .ps1 文件添加 UTF-8 BOM（EF BB BF）
- build-release.ps1 新增 Step 3.45 编码规范化：staging 中 .ps1 自动补 BOM，.cmd 验证纯 ASCII 无 BOM

### 4.2 .cmd 启动器中文乱码（P0）

**问题**: 早期 .cmd 文件包含中文标题、框线字符。中文 Windows CMD 按 GBK 解析 UTF-8 无 BOM 文件，出现 `'乱码echo' 不是内部或外部命令`。

**修复** (`5e19269`):
- 全部 6 个 .cmd 改为纯 ASCII（无中文、无 emoji、无框线）
- 中文提示全部留在 .ps1 中
- scripts/check.ps1 新增 .cmd BOM/ASCII 检查

### 4.3 git archive 跳过 API Key 扫描（P0）

**问题**: build-release.ps1 在检测到 .git 时优先使用 `git archive` 打包，直接生成 ZIP 不经过 staging，跳过 API Key 内容扫描。

**修复** (`595b129`):
- 删除 git archive 分支
- 统一走 staging 目录流程：Copy-Filtered → API Key 扫描 → ZIP 创建

### 4.4 API Key 扫描作用域 bug（P0）

**问题**: 函数内 `$apiKeyHits += @{...}` 因 PowerShell 函数作用域创建了局部变量，外层 `$apiKeyHits.Count` 始终为 0，扫描到真实 Key 也不会中止。

**修复** (`1faf53f`):
- `$apiKeyHits` 从 `@()` 改为 `[System.Collections.Generic.List[object]]::new()`
- 函数内用 `.Add()` 替代 `+=`

### 4.5 安装命令超时过短（P0）

**问题**: `Invoke-CommandSafe` 默认 60s 超时。Native Install、winget、npm install 在慢网/低配机器上可能超过 60s，误判失败。

**修复** (`49c6fee`):
- 安装类调用显式传 TimeoutSec：下载 180s、执行 600s、winget 900s、npm 900s

### 4.6 release 打包 exclude-list 风险（P0）

**问题**: 早期排除模式可能漏掉临时文件、草稿文档、截图等未预见的新文件。

**修复** (`49c6fee` + `105f5d9`):
- 改为 allow-list：只允许白名单文件进入 release
- docs/ 和 examples/ 文件级白名单（不是整目录）
- 白名单条目缺失时 exit 1（不是 WARN 跳过）
- 新增源目录 API Key 预扫描（allow-list 过滤前）

### 4.7 ZIP 内直接双击 .cmd（P0）

**问题**: 用户在 ZIP 内直接双击 .cmd，找不到 Start-Here.ps1 或 lib/bootstrap.ps1。

**修复** (`49c6fee`):
- 6 个 .cmd 在执行前检查关键文件存在性
- 缺失时输出英文提示 "Please extract the full ZIP package first"

### 4.8 控制台 emoji/框线乱码风险（P0）

**问题**: ✅❌⚠╔═║━┌│ 等字符在老终端/远程工具/缺字体环境显示为方块。

**修复** (`105f5d9`):
- 全部 7 个 .ps1 文件移除 emoji 和框线字符
- 统一使用 `[OK]`/`[WARN]`/`[ERROR]`/`[SKIP]`/`[INFO]`
- 框线改为 `===` 和 `---`

### 4.9 Native Install 后 PATH 未刷新（P0）

**问题**: 安装器已成功写入 PATH，但当前 PowerShell 进程未刷新，`Test-ClaudeInstalled` 检测不到 claude，脚本误判失败。

**修复** (`105f5d9`):
- common.ps1 新增 `Refresh-CurrentProcessPath`（合并 Machine + User PATH）
- Native Install/npm 后检测失败时先刷新 PATH 再重试
- 最终摘要区分"已安装"和"可能已安装（PATH 未刷新）"

### 4.10 install.ps1 版本号未更新（P1）

**问题**: install.ps1 中 `$ScriptVersion = "1.2.0"` 未随 v1.3.0 更新。

**修复** (`de6e9fe`): 改为 `"1.3.0"`。

## 五、当前版本关键设计决策

### 5.1 Claude Code 安装策略

1. 优先 Native Install (`irm https://claude.ai/install.ps1 | iex`)
2. Native Install 失败 → npm fallback (`npm install -g @anthropic-ai/claude-code@latest`)
3. Node.js < 18 时尝试 winget 安装 `OpenJS.NodeJS.LTS`（含 `--accept-source-agreements`）
4. 已有 Claude Code 时检测版本，询问是否更新，默认不强制

### 5.2 WSL 策略

- 检测但不默认安装/启用
- 无 WSL 时只提示「高级选项」
- 有 WSL Ubuntu 时询问是否配置
- WSL 配置由 install_wsl.sh 负责

### 5.3 安全设计

| 层级 | 措施 |
|------|------|
| 输入 | SecureString 不回显 |
| 存储 | 仅写入本机 `%USERPROFILE%\.claude\settings.json` |
| 日志 | Mask-ApiKey 脱敏（前 4 + 后 4） |
| 报告 | 安装报告/诊断报告不含完整 Key |
| 传输 | API 测试仅调 DeepSeek 官方接口 |
| 打包 | 源预扫描 + staging 扫描双重 API Key 检测 |
| 发布 | allow-list + 禁入条目硬失败 + 编码校验 |

### 5.4 发布流程

```powershell
# 1. 打包
powershell -ExecutionPolicy Bypass -File .\scripts\build-release.ps1 -Version "1.3.0"

# 2. 泄露测试
Set-Content .\leak-test.md "DEEPSEEK_API_KEY=sk-leaktest0123456789abcdef0123456789"
powershell -ExecutionPolicy Bypass -File .\scripts\build-release.ps1 -SkipSha256
Remove-Item .\leak-test.md

# 3. 验收（见 docs/发布前验收清单.md）
```

- 依赖 DeepSeek 官方 Anthropic 兼容接口的稳定性
- 多模态功能取决于 DeepSeek 官方支持

## 八、售后话术要点

1. 用户第一屏永远是「解压 → 双击开始安装.cmd」
2. 输入 API Key 时不显示是**安全保护**，不是 bug
3. API 测试失败 ≠ 安装失败（先检查 Key + 余额 + 网络）
4. 诊断报告不含完整 Key，可以放心发给卖家
5. 不要发送 API Key 给任何人
6. DeepSeek 5xx 错误是官方服务问题，不是配置问题
7. 建议用户关闭终端窗口重新打开后再运行 claude
8. 中文文件名异常时用 `Start-Install.cmd`
